## 1. V1 目标与边界

### 1.1 产品定位

- 路线：先自用再商业化（mixed）
- V1 策略：均衡版（覆盖核心模块，深度适中）
- 发布方式：单人直发（无人工评审流）
- 质量策略：保留轻量自动质检
- 版本策略：自动版本号（v0.1/v0.2...）+ 一键回滚

### 1.2 V1 核心价值目标

- 将“想法 -> 可发布需求初稿”缩短到 10-20 分钟
- 打通“立项 -> 成文 -> 执行 -> 可视化”核心链路
- 保持低流程复杂度，优先保证可用性和体验

### 1.3 V1 非目标（不在本版）

- 多角色审批工作流
- 深度第三方集成（Jira/Git/企业微信机器人）
- 复杂多租户计费体系

## 2. 旅程1冻结：立项到需求初稿发布

### 2.1 旅程定义

用户从立项入口进入，完成项目信息输入，生成近可发布需求初稿，通过自动质检后直接发布为新版本。

### 2.2 双入口并列（等权）

1. AI 优先入口  
   - 输入一句话/一段话项目描述
   - 系统自动生成立项字段与需求初稿

2. 表单优先入口  
   - 先填核心字段（名称、类型、目标、时间等）
   - AI 基于结构化字段补全初稿

两个入口最终进入同一工作台，使用同一套编辑、质检、发布、版本机制。

### 2.3 旅程字段（冻结）

#### 2.3.1 核心字段（必填）

- 项目名称
- 项目描述
- 项目类型
- 优先级
- 起止日期

#### 2.3.2 增强字段（选填，AI 可补全）

- 项目背景
- 目标用户
- 业务价值
- 范围边界（含不做项）
- 关键里程碑
- 风险与假设

### 2.4 初稿发布标准（near-final）

初稿必须具备以下章节且有可评审正文：

- 背景与目标
- 范围与边界
- 用户与场景
- 功能需求（模块级）
- 非功能需求（性能/安全/稳定）
- 验收标准
- 风险与假设

### 2.5 轻量自动质检（发布前）

1. 结构完整性检查  
   - 必要章节缺失时阻止发布并提示补齐

2. 术语一致性检查  
   - 核心实体命名冲突提示（如 Story/任务/模块）

3. 基础矛盾检查  
   - 日期逻辑冲突  
   - 范围与不做项冲突  
   - 目标指标自相矛盾

### 2.6 发布与版本

- 发布动作：点击发布即生成新版本
- 版本号：自动递增（v0.1 -> v0.2）
- 变更说明：自动生成摘要（可人工编辑）
- 回滚能力：一键回滚到任意历史版本

### 2.7 旅程1验收标准

- 从立项到初稿发布全程 <= 20 分钟
- AI 首字响应 <= 3 秒
- 自动质检可识别缺章、术语冲突、明显矛盾
- 发布后可见历史版本且可成功回滚

## 3. 轮次1：项目管理 + 文档AI协作（立项-成文闭环）

### 3.1 项目管理模块需求

- 创建/编辑/查看项目
- 项目状态管理（草稿、进行中、完成、归档）
- 项目模板（基础模板）
- 项目健康度卡片（解释项）
- 风险卡片（可直接写入需求文档）

### 3.2 文档AI协作模块需求

- AI 对话生成文档
- 三段式生成流程：大纲 -> 草稿 -> 审校建议
- 文档编辑与保存
- 文档版本管理、版本对比、回滚
- 自动生成变更摘要

### 3.3 轮次1验收标准

- 项目可在 3 分钟内完成创建
- 可从项目页一键进入文档工作台
- 需求初稿章节完整，且可直接发布
- 发布后版本记录可追溯，回滚成功率 100%

---

## 4. 轮次2：Story/Task/看板（需求-执行闭环）

### 4.1 Story 管理

- 从需求文档中生成 Story 草案
- Story 必填验收标准（Acceptance Criteria）
- Story 与模块/需求章节关联

### 4.2 Task 管理

- Story 拆分 Task
- 任务状态流转（To Do/In Progress/Done）
- 任务优先级与负责人

### 4.3 看板能力

- 看板视图展示
- 拖拽流转状态
- WIP 限制与阻塞原因记录

### 4.4 轮次2验收标准

- Story 至少 80% 可由模板/AI草案直接复用
- Task 状态更新 <= 2 次点击
- 阻塞任务可被识别并进入风险卡片
- 看板数据与项目进度统计一致

---

## 5. 轮次3：Dashboard + 人员负荷（执行-决策闭环）

### 5.1 Dashboard

- 执行视图（今日）
- 管理视图（趋势）
- 关键指标：进度、延期风险、吞吐、阻塞数
- AI 周报（问题-原因-行动项）

### 5.2 人员负荷

- 成员任务负荷展示
- 简单负荷预警（高/中/低）
- 调整建议（建议转移任务候选）

### 5.3 轮次3验收标准

- Dashboard 核心页面加载 <= 2 秒
- 项目经理 5 分钟内可完成周会准备
- AI 周报可生成明确行动项与优先级
- 高负荷成员可被系统自动标记

## 6. V1 非功能基线（冻结）

- 页面体验：核心页面加载 < 2 秒
- 接口体验：普通 API P95 < 800ms
- AI 体验：首字 < 3 秒
- 稳定性：目标可用性 99.5%
- 安全性：基础 RBAC、审计日志、敏感信息保护

## 7. 迭代0-4 初步排期与里程碑

### 7.1 迭代规划

1. 迭代0（1-2周）  
   - 账号权限、基础数据模型、项目骨架、日志与版本底座

2. 迭代1（2周）  
   - 项目管理 + AI 文档生成 + 自动版本与回滚

3. 迭代2（2周）  
   - Story/Task/看板 + 基础分配策略

4. 迭代3（2周）  
   - Dashboard + 负荷预警 + AI 周报 + 导出

5. 迭代4（1-2周）  
   - 知识库增强 + 体验优化 + 性能稳定性加固

### 7.2 里程碑定义

- M0：完成基础底座（迭代0结束）
- M1：完成立项-成文闭环（迭代1结束）
- M2：完成需求-执行闭环（迭代2结束）
- M3：完成执行-决策闭环（迭代3结束）
- M4：形成可长期使用的 V1 稳定版（迭代4结束）

---

## 8. 执行规则（避免需求反复）

- 规则1：按轮次冻结需求，不跨轮次混改
- 规则2：每轮先定用户旅程，再定字段和验收
- 规则3：需求变更只允许进入下一轮，不回滚当前轮
- 规则4：所有新增需求必须明确商业价值与使用频次
