# REQ-M06 变更请求（CR）与需求联动（v1.0）

---

## 1. 模块定位

开发过程中会 **随时** 出现范围/规则/接口/库表等变更。本模块提供 **轻量 CR（Change Request）台账**：**谁提、审什么、何时采纳、改了哪些工件、跟哪些 Task**，与 **REQ-M02 文档版本**、**REQ-M04 执行** 打通，**不搞复杂配置台**，交互以 **列表 + 详情 + 关联跳转** 为主。

> **原则**：**正文仍以文档编辑与版本 Diff 为准**；CR 提供 **业务语义与追溯**（为何而改、何时生效、是否废止旧内容），避免在正文里手工维护一套「第二套版本系统」。

---

## 2. 与上下游模块的关系

| 模块 | 关系 |
|------|------|
| REQ-M02 / M02B | CR **采纳**后，用户在对应文档中改内容并 **保存**；保存时可 **勾选关联本 CR**（版本历史条目带 CR 编号） |
| REQ-M02C / M02D | CR 可标记「影响接口/库」；具体修改在接口/库模块完成，CR 详情中 **登记链接或说明** |
| REQ-M03 / M04 | 采纳后可 **一键生成跟进 Task** 或 **关联已有 Task** |
| REQ-M04 | Task 详情可展示 **关联 CR**（只读跳转） |
| REQ-M07 | 缺陷可 **关联 CR**；范围问题 **提示走 CR**，不自动建单 |

**CR 贯穿项目生命周期**，不限于「排期之后」；流程图中与 M04 **并行存在**。

---

## 3. CR 单据（V1 字段）

| 字段 | 说明 |
|------|------|
| 编号 | 项目内唯一，如 `CR-001`（自动递增） |
| 标题 | 一句话 |
| 描述 | 背景、期望、验收要点（富文本/Markdown 均可，V1 可纯文本） |
| 提出人 / 提出时间 | 自动 + 可改 |
| 状态 | **草稿 → 待评审 → 已采纳 / 已拒绝 → 已实现 → 已关闭**（见 §4） |
| 影响类型（多选） | 需求文档 / 技术设计 / 接口 / 库表 / 其它（文本） |
| **废止说明**（可选） | 自由文本：**明确写明「废止哪些旧条款」**（如「原模块 X §3.2 登录流程作废」）；**不强制**段落级锚点（V1 不做行号绑定，降低实现成本） |
| **生效说明**（可选） | 自由文本：**自何时起按新需求执行**（可与「关联文档版本」二选一或并存） |

---

## 4. 状态机（V1，宜简单）

```
草稿 ──提交评审──► 待评审 ──通过──► 已采纳 ──开发完成──► 已实现 ──验收──► 已关闭
                      │
                      └──驳回──► 已拒绝（可附原因，可重新编辑为草稿）
```

- **已采纳**：表示 **范围上同意变更**；**不自动改文档**，避免静默覆盖。
- **已实现**：由负责人勾选或关联「相关 Task 均已完成」（V1 可 **手工标记**，不做强制校验）。
- **已关闭**：归档；列表默认可筛「未关闭」。

---

## 5. 与需求/设计文档的联动（实现简单、交互清晰）

### 5.1 保存文档时关联 CR（推荐主路径）

- 在 **REQ-M02 / M02B** 等文档编辑器点击 **保存**（或「发布版本」）时，提供可选字段：**关联 CR**（下拉选「已采纳」状态的 CR，可多选）。
- 该次保存产生的 **版本历史记录** 展示：`关联 CR：CR-001, CR-003`。
- **追溯**：打开 CR 详情 → **「相关文档版本」** 列表：文档名、版本号、保存时间、操作者、**跳转 Diff**。

### 5.2 正文内标注（可选、不强制）

- 用户可在正文中自行增加 **简短说明行**（非系统自动插入），例如：  
  `【CR-003 | 2025-03-22 起生效】本节替换原登录流程。`  
  `【CR-003 | 原 §3.2 已废止，见 CR 详情废止说明】`  
- **系统不解析**这些标签做自动状态机（V1）；**真相源**仍是 **版本 Diff + CR 台账**。

### 5.3 「何时加入 / 何时作废」怎么回答（产品口径）

| 问题 | V1 答案 |
|------|---------|
| 新需求何时写入？ | **文档某版本的保存时间** + 可选 **关联 CR**；CR 内 **生效说明** 可写「自迭代 2 起」等业务语义 |
| 旧内容何时作废？ | **废止说明**（CR 内）+ **Diff**（看删除了什么）；若需「对外一句话」，写在 **生效/废止说明** 或正文自选标注行 |
| 不想写废止段落？ | 允许；仅依赖 **Diff** 与 CR 标题/描述亦可 |

## 6. 与接口（M02C）、库（M02D）

- CR 详情：**勾选「影响接口/库」** 后，可 **添加链接**（到具体接口定义或表）或 **纯文本说明**。
- **实际字段变更**仍在 M02C/M02D 内完成；CR 起 **台账与跳转** 作用。

---

## 7. 与 Task（M04）

- **已采纳** 的 CR：按钮 **`生成跟进 Task`**（生成一条或多条草案 Task，进入 M03/M04 约定流程）或 **`关联已有 Task`**。
- Task 详情展示 **关联 CR**（只读），点击进 CR。

---

## 8. 列表与入口

- 项目左侧或 **`变更 / CR`**：**全部 CR 列表**（筛选：状态、提出人、时间）
- **迭代看板 / Story**（可选 V1.5）：小徽章「本迭代含 N 个未关闭 CR」（不做强依赖）

---

## 9. AI（按需，与全站一致）

- 抽屉：`根据 CR 描述建议应改哪些文档章节、接口、Task 清单` → 输出 **建议列表**，用户 **手动采纳** 到 CR 描述或自行去改文档；**不自动落库文档**（与 M02 安全策略一致）。

---

## 10. 验收标准（产品口径）

- 可创建 CR，状态按 §4 流转，**已拒绝 / 已关闭** 可区分
- 文档保存时可 **可选关联 CR**，版本历史 **可见关联**
- CR 详情可维护 **废止说明 / 生效说明**，可 **关联 Task** 或 **生成跟进 Task**
- 从 CR **可跳转到文档版本或 Diff**（至少文档名+版本号+链接）

---

## 11. 明确不做（V1 砍 scope）

- 正文 **自动**插入/同步 CR 标签、段落级 **强制** 废止锁
- 与外部 Jira/Azure DevOps **双向同步**
- 复杂审批流（多级、会签）；V1 **单级「待评审→通过/驳回」** 即可，或用 **角色权限** 控制谁可点「通过」
