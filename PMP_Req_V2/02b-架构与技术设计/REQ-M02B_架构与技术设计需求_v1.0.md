# REQ-M02B 架构与技术设计需求（v1.0）

---

## 1. 模块定位

本模块承接 **需求文档（REQ-M02）达到可开发基线之后** 的技术侧工作，并 **贯穿后续开发过程**（可与迭代并行）：

- 冻结或确认 **技术栈、系统架构、关键设计决策**
- 产出 **技术类文档**（可与需求侧同样采用「总纲 + 分文档」结构）
- **接口设计文档、数据库设计文档** 允许在 **开发过程中持续补全**；在需求已确认的前提下，优先支持 **AI 辅助生成初稿**，便于 Task 拆分时前后端 **接口名与参数一致**
- 设计确认后，通过按钮将内容 **同步为结构化数据**，详见独立模块：**REQ-M02C 接口管理**、**REQ-M02D 数据库结构管理**（含接口/表字段 CRUD、DDL 导出）
- 与需求模块 **双向可追溯**（需求章节 ↔ 设计章节/模块）
- 为 **Story/Task 拆分、联调、测试范围** 提供依据

> 说明：本文件只描述**产品功能与交互需求**，不规定具体技术实现细节。

## 2. 与上下游模块的关系

| 上游 | 本模块 | 下游 |
|------|--------|------|
| REQ-M01 立项（含「技术栈/人力后续补全」勾选） | 在此由开发负责人等补全并固化 | REQ-M03 迭代与 Story |
| REQ-M02 总纲 + 分模块需求文档 | 设计文档与需求文档建立关联 | REQ-M02C 接口管理、REQ-M02D 数据库结构管理、REQ-M03 迭代与 Story、REQ-M04 Task |

- 立项时若勾选 **「技术栈后续由技术负责人补全」**，对应待办在本模块工作台 **首屏提示**，直至关键字段补齐或用户显式标记「暂不填写」（见 7.2）。

## 3. 文档体系（建议结构）

### 3.1 与技术侧对齐的「总纲 + 分文档」

与需求侧（REQ-M02）一致，技术侧采用：

- **L1 总技术文档（Master）**  
  - 技术栈总览、系统上下文、架构风格、跨模块原则、非功能约束（性能/安全/可用性）、环境与部署策略（可简写）
- **L2 分文档（按模块或按域）**  
  - 与需求 **模块文档** 建议 **一一对应**（同名或显式关联 ID），便于检索与 Diff  
  - 每份可包含：模块边界、核心组件、数据与接口边界、关键流程、风险与待定项

### 3.2 可选文档类型（V1 可裁剪）

| 类型 | 说明 | V1 建议 |
|------|------|---------|
| 技术栈说明 | 前后端、DB、中间件、运行环境 | 建议必有（可合在总文档内） |
| 架构设计书 | 上下文、分层、部署视图 | 建议必有（可简版） |
| 详细设计 / 模块设计 | 与需求模块对齐 | 建议按模块各一份，可渐进完善 |
| 接口设计 | API 路径、方法、名称、请求/响应参数、错误约定 | **允许开发中补齐**；需求确认后可用 **AI 根据需求+模块生成初稿** |
| 数据库设计 | 表、字段、类型、索引、关系（逻辑/物理） | **允许开发中补齐**；需求确认后可用 **AI 生成初稿**，与接口设计 **对齐命名** |

## 4. 页面与交互

### 4.1 技术设计工作台（主页面）

- **左侧**：文档树（总技术文档 + 各模块技术文档，结构与需求文档树 **并列或嵌套展示「需求↔设计」关联」**）
- **中间**：当前文档编辑区（Markdown / 富文本，与 REQ-M02 一致）
- **右侧**：**AI 辅助抽屉**（默认收起；**交互口径** 见 **REQ-M11**，**站级壳布局** 见 **REQ-M09**）

### 4.2 创建流程

1. **从需求进入**：在需求工作台提供入口 **「生成技术文档骨架」**  
   - 依据已有 **模块列表** 自动创建：1 个总技术文档 + N 个模块技术空文档（带章节模板）
2. **手动创建**：负责人手动新增模块技术文档或章节

### 4.3 与需求的联动

- 在 **模块需求文档** 侧可显示 **「关联技术文档」** 链接；在技术文档侧显示 **「关联需求文档」**
- 支持 **从需求章节跳转** 到对应模块技术文档（按模块 ID 绑定）

### 4.4 版本、Diff、AI 撤回（与 REQ-M02 一致）

- **所有技术类文档**均具备：**完整版本历史**、**任意两版 Diff**、**回滚产生新版本**
- **AI 写入正文**支持 **一键撤回**，且 **清除相关 AI 上下文**（等价于对该文档重新开问）
- **模块技术文档删除**：与需求模块文档策略一致——**第一版硬删 + 二次确认**（不设回收站）

### 4.5 回写总技术文档摘要

- 从 **分模块技术文档** 汇总到 **总技术文档** 的摘要时：**保存前必须弹窗确认**（可展示将写入的要点或 Diff），用户确认后再落库并生成总文档新版本

### 4.6 技术负责人与项目首页（与 REQ-M01 对齐）

- **技术负责人立项时可不填**。
- **项目首页**：未选定技术负责人时，展示 **非阻塞提示**（如横幅）：「尚未选定技术负责人」，并提供 **去指定** 入口（指定后提示消失）。
- 指定技术负责人 **不强制** 作为进入其它阶段的门槛（与 V1 低门槛策略一致）。

### 4.7 接口设计文档与接口管理（独立模块）

- **接口设计文档** 在本模块（技术设计工作台）中维护：版本、Diff、AI 协作、一键撤回等规则同本文 §4.4、§5。
- **结构化接口库、同步规则、列表与 CRUD** 见独立文档：**[REQ-M02C_接口管理需求_v1.0.md](../02c-接口管理/REQ-M02C_接口管理需求_v1.0.md)**。
- 技术工作台提供入口按钮 **`生成/同步到接口管理`**（行为以 REQ-M02C 为准）。

### 4.8 数据库设计文档与数据库结构管理（独立模块）

- **数据库设计文档** 在本模块中维护。
- **表/字段结构化数据、同步规则、CRUD、DDL 一键生成（SQLite/SQL Server/MySQL）** 见独立文档：**[REQ-M02D_数据库结构管理需求_v1.0.md](../02d-数据库结构管理/REQ-M02D_数据库结构管理需求_v1.0.md)**。
- 技术工作台提供入口按钮 **`生成/同步到数据库结构管理`**（行为以 REQ-M02D 为准）。

## 5. AI 协作（按需）

> 全站统一见 **REQ-M11**。

- **触发**：用户点击 `AI辅助` 后展开抽屉；不在用户未操作时主动打断
- **典型场景**：
  - 根据需求模块摘要生成技术方案草案
  - 补充非功能与风险清单
  - 根据技术栈生成「环境与依赖」检查清单
  - **根据已确认需求生成接口设计初稿、数据库设计初稿**（与模块边界对齐）
- **落点**：先 **建议区**，**应用** 后写入正文；与 REQ-M02 相同

| 角色 | 主要职责 |
|------|-----------|

## 7. 业务规则（V1）

### 7.1 进入迭代规划（下游）

- V1 **不设强制门槛**：即使技术文档未写完，也允许进入迭代规划（与「需求提交即立项」理念一致）
- **系统提示（非阻塞）**：若总技术文档或关键模块技术文档为空，进入迭代时 **顶部横幅提示**「建议先完善技术设计，降低返工风险」
- **接口 / 库表** 未同步到 **REQ-M02C / REQ-M02D** 时：**不阻塞** Task 创建，但在 **拆分涉及前后端联调的任务** 时 **非阻塞提示**「建议先同步接口定义，避免命名不一致」

### 7.2 立项待办闭环

- 立项勾选「后续补全技术栈」时：在本模块 **待办列表** 中展示；用户可 **标记已完成** 或 **暂不填写（需填简短原因）**

### 7.3 文档命名建议

- 总文档：`ARCH-MASTER_{项目名}_vX.Y`
- 模块文档：`ARCH-MOD-{模块编号}_{模块名}_vX.Y`

## 8. 验收标准（产品口径）

- 可从需求阶段 **一键生成** 技术文档树骨架（总 + 分模块）
- 技术文档与需求模块 **可双向跳转、关联可见**
- 技术文档具备与需求文档 **同等能力的版本历史、Diff、回滚**
- AI 辅助 **不抢占主界面**；一键撤回 AI 修改且 **清除相关上下文** 可用
- 分文档 → 总文档摘要回写 **必经确认**
- 模块技术文档删除：**二次确认后硬删**
- **接口设计 / 数据库设计** 可在开发中编辑；需求确认后可 **AI 生成初稿**
- **结构化接口库、表结构、同步、CRUD、DDL** 的验收见 **REQ-M02C、REQ-M02D**
