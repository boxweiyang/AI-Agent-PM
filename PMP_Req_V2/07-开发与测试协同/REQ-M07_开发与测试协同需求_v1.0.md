# REQ-M07 开发与测试协同需求（v1.0）

---

## 1. 模块定位

在 **REQ-M04** 的 Task 看板（含 **待测试** 列）之上，补齐 **开发完成 → 测试验证 → 缺陷闭环 → 必要时回退开发** 的 **轻量协作**，并与 **接口（M02C）**、**变更（M06）** 可跳转关联。

**本模块侧重**：

1. **提测**：把「可以测了」说清楚（范围、环境、备注），**不替代** CI/CD 平台。  
2. **缺陷（Bug）**：独立单据、状态清晰、**默认关联 Task**（或 Story/迭代），便于统计与追溯。  
3. **联调**：以 **接口清单 + Task** 为索引的协作视图（过滤与跳转），**不强制**单独「联调工单类型」。

**不做（V1 明确砍 scope）**：完整用例库、自动化执行平台、性能测试平台、与外部缺陷系统双向同步。

## 4. 提测（Submit for Test）

### 4.1 触发

- **入口 A**：Task 详情 **`发起提测`**（生成一条 **提测记录**，并预填关联 Task、负责人、迭代）。  
- **入口 B**：项目 **`测试协同 → 新建提测`**（手工选多个 Task 或选 Story/迭代范围）。

### 4.2 提测记录字段（V1）

| 字段 | 说明 |
|------|------|
| 编号 | 如 `ST-001` |
| 关联范围 | 多选 Task；可选填 **迭代 / Story**（与 M03 一致） |
| 环境说明 | 文本：部署地址、分支、构建号等（**手工填写**，不接流水线） |
| 测试要点 | 文本：建议测什么、风险点 |
| 状态 | **待测试 → 测试中 → 已通过 / 已驳回** |
| 创建人 / 时间 | 自动 |

### 4.3 规则

- **已驳回**：须填 **驳回说明**；相关 Task 可 **建议** 拖回 `进行中`（**系统提示**，不强制改状态）。  
- **已通过**：可 **提示** 将范围内 Task 拖向 `已完成`（**不自动**，避免误关闭）。

## 5. 缺陷（Bug）

### 5.1 缺陷单字段（V1）

| 字段 | 说明 |
|------|------|
| 编号 | 如 `BUG-001` |
| 标题 / 描述 | 现象、复现步骤（文本；**附件 V2**） |
| 严重程度 | **致命 / 高 / 中 / 低**（四档即可） |
| 状态 | 见 §5.2 |
| 关联 | **首选关联 Task**；可选 Story、迭代、**接口（M02C）**、**表/库（M02D，可选）** |
| 发现人 / 指派给 | 人力池成员（与 M05 同源） |
| 关联 CR | 可选；用于「此缺陷源于某次变更」类说明 |

### 5.2 缺陷状态机（V1）

```
新建 ──确认有效──► 打开 ──开发修复──► 已修复 ──测试验证──► 已关闭
  │                    │
  └──重复/非问题──► 已拒绝          └──验证失败──► 重新打开（回到 打开）
```

- **新建**：测试或任何人提交。  
- **打开**：经 **测试/负责人** 确认有效（V1 可简化为 **提交即打开**，跳过「新建」人工确认）。  
- **已修复**：开发标记；进入 **待验证** 语义（可与状态名合并为 **待验证** 若需减少状态数）。  

**状态精简版（可选实现）**：`打开 / 待验证 / 已关闭 / 已拒绝` 四态亦可，产品在实现前二选一，本文以 **完整版** 为口径，**允许裁剪**。

### 5.3 与 Task 的联动

- 从缺陷 **`创建跟进 Task`**（类型建议 **缺陷修复**）：进入 M04，**自动关联本缺陷**。  
- Task 详情展示 **关联缺陷** 列表（只读跳转）。  
- 缺陷 **关闭** 不自动关闭 Task；由用户在 M04 合并或关闭修复 Task。

### 5.4 与 CR（M06）

- 若缺陷根因是 **需求/范围错误**：关闭或处理缺陷时，**勾选「建议登记 CR」**（仅 **跳转/提示**，不自动创建 CR）。  
- 若已有 **已采纳 CR**：可 **关联 CR**，便于 Dashboard 与审计。

## 6. 联调协作视图

### 6.1 目标

前后端围绕 **同一接口（M02C）** 对齐：**谁在做、哪个 Task、有无未关闭缺陷**。

### 6.2 页面（V1）

- 项目 **`联调`**：表格或列表 = **本迭代（可选）下，已关联接口的 Task**  
  - 列示例：接口标识、Task、负责人、状态、**未关闭缺陷数**、跳转 M02C  
- **不单独**维护「联调单」实体（降低实现成本）；若后续需要再升维。

---

## 7. AI 辅助（按需，与全站一致）

- **提测**：`根据所选 Task + 接口摘要生成测试要点草稿` → 用户编辑后保存。  
- **缺陷**：`根据描述补全复现步骤模板`（可选）。  
- 均为 **抽屉 + 应用**，不自动落库不经确认。

---

## 8. 列表与权限（V1）

- **`测试协同`** 下挂：**提测列表**、**缺陷列表**（Tab 或子菜单）。  
- **项目负责人 / 测试角色** 可改缺陷指派与状态；**开发** 可标记已修复；具体 **角色与 M01 对齐**，不在本文展开细则。

---

## 9. 验收标准（产品口径）

- 可从 Task **发起提测**，提测单有 **环境说明、范围、状态**  
- 可创建 **缺陷**，关联 **Task/接口**，状态可闭环至 **已关闭/已拒绝**  
- **联调** 视图能按 **接口 + Task** 聚合展示并跳转 M02C  
- 与 M04 看板 **无重复强迫操作**（状态仍以 M04 为准，M07 以单据与提示辅助）

---

## 10. 建议全流程补句（产品叙述）

```
开发将 Task 拖至「待测试」或发起提测 → 测试基于提测记录执行 → 发现问题建缺陷并指派
→ 开发修复并标记已修复 → 测试验证后关闭缺陷；必要时关联 CR 或新建跟进 Task。
```
